## Foglio1

| Quesito | Risposta |
| --- | --- |
| 1.  Il progetto deve partire da una reference implementation ufficiale del CSI Piemonte (es. uno skeleton di progetto Spring Boot + Angular già configurato per ECaaS/Kubernetes), oppure lo scheletro applicativo deve essere costruito ex-novo dal team di sviluppo? La disponibilità di un template precostituito impatta significativamente sui tempi della fase di setup iniziale.  | allego alla mail le linee guida Cloud Native per i fornitori |
| 2. La versione Angular è indicata come 19+ con stato [PROPOSTA]. Esiste una versione specifica già validata e approvata dal CSI Piemonte nel catalogo ECaaS, oppure il team deve usare l'ultima stabile disponibile al momento dell'avvio?  | La libreria Angular v. 19.x è in stato CURRENT ed è utilizzata nello stack applicativo Single Page Applications Angular2/springboot/Resteasy, v.2.1.0 |
| 3. Sono disponibili una Component Library o uno UI Kit ufficiale del CSI Piemonte (es. basato su Angular Material o Bootstrap) da utilizzare per garantire la coerenza grafica con gli altri applicativi regionali? In caso affermativo, indicare il repository o il pacchetto npm.  | non c'è UI kit ufficiale ma ci sono dei componenti QUASAR |
| 4. L'accesso tramite SPID/CIE richiede l'integrazione con un Identity Provider (IdP) specifico del CSI Piemonte o con un servizio di federazione centralizzato? È disponibile un SDK o una libreria da integrare nell'applicativo Angular?  | Identity provider del CSI-Piemonte, GASP Salute |
| 5. La versione Spring Boot è indicata come 3.x. È richiesta una versione minore specifica (es. 3.2, 3.3)? Esistono vincoli di compatibilità con librerie interne CSI già in uso?  | la pila tecnologica standard e current in CSI è sprigboot3_k8s_v1.0.0 è indicato di attestarsi almeno alla v. 3.4.10 per mettersi al riparo da vulnerabilità note |
| 6. È richiesto un API Gateway centralizzato del CSI Piemonte come punto d'ingresso per le API esposte, oppure l'API Gateway è un componente da implementare all'interno del progetto (es. Spring Cloud Gateway)?  | Per questo progetto l'API Gateway CSI non sarà utilizzato, optando per un'integrazione diretta.  Per procedere, è necessario ricevere il file Swagger (OpenAPI Specification), che dovrà includere: Definizione degli Endpoint: Elenco completo delle risorse e dei metodi (GET, POST, etc.). Modelli Dati: Struttura dettagliata dei messaggi di richiesta e risposta. Standard di Sicurezza: Specifiche sui meccanismi di autenticazione (es. API Key, OAuth2, Basic). Gestione Errori: Mapping dei codici di stato HTTP e relativi messaggi di errore. Una volta ricevuto lo Swagger, potremo validare la compatibilità tecnica e avviare lo sviluppo dei connettori |
| 7.  I servizi REST di AURA e Gestione Deleghe utilizzano quale protocollo di autenticazione inter-servizio? (es. OAuth2 Client Credentials, Mutual TLS, API Key). Sono disponibili WSDL/OpenAPI degli endpoint di AURA e del servizio Deleghe da usare durante lo sviluppo?  | AURA servizi SOAP. Per l’autenticazione è necessario inserire nell’header della richiesta uno UsernameToken con password type text, fornito dal gruppo di assistenza anagrafiche, che provvede a censire le credenziali sulla nostra piattaforma IRIS per ciascun fruitore. I WSDL, ci sono è necessario solo specificare quali servizi è necessario invocare, così da poter fornire i relativi WSDL ed endpoint Deleghe servizi SOAP , autenticazione OAuth2 (disponibili i wsdl) |
| 8. Il Notificatore Regionale espone un'API REST o SOAP? È disponibile la documentazione tecnica dell'interfaccia? Quale canale viene usato per le notifiche al cittadino (email, SMS, notifica push su app IO)?  | i Canali a disposizione sono email, push e io. SMS è disponibile ma di solito non utilizzato. È dispobile anche il canale mex che inviare le notifiche all'area riservata del portale Salute Piemonte. La documentazione aggiornata si trova a questa url https://gitlab.csi.it/user-notification-platform/unpdocumentazione/-/blob/master/documentazione%20fornitori/NOTIFY-specifiche.md?ref_type=heads |
| 9. I SIA delle ASR che ricevono le notifiche del BATCH-01 espongono ancora endpoint SOAP (come da Specifica-WebService_ConsensoRegionaleAziendale_v03 AS-IS) oppure è prevista una migrazione a REST/JSON nel TO-BE? In caso di SOAP, il backend deve includere un client SOAP (es. Apache CXF) — in caso di REST, è sufficiente RestTemplate/WebClient.  | I servizi delle ASR non devono cambiare |
| 10. Esiste una versione di PostgreSQL Community K8S già certificata e disponibile sull'infrastruttura ECaaS in produzione? Il provisioning avverrà tramite Helm Chart CSI standard o tramite un operatore Kubernetes (es. CloudNativePG)?  | I progetti su Ecaas usano dei DBAAS lato account Nivola. Il provisioning avviene con scheda di provisioning "standard".  |
| 11. Migrazione dati: Il passaggio da PostgreSQL 9 a PG17 richiede un percorso di migrazione pianificato (upgrade a step intermedi o dump/restore), non è un aggiornamento automatico sul posto. È già stata pianificata e approvata una strategia di migrazione dei dati dal database PostgreSQL AS-IS a PostgreSQL 17 TO-BE? Chi è il responsabile della redazione del piano di migrazione (CONSPREF-DMP)?  | al momento non è ancora stato formalizzato un piano di migrazione |
| 12. Alcune sintassi e funzionalità di PG9 sono cambiate o deprecate in PG17 (es. gestione delle sequenze, funzioni to_timestamp, comportamento di NULL in indici). È previsto un'attività di audit della struttura AS-IS per identificare incompatibilità prima dello sviluppo?  | sì questa attività è da prevedere |
| 13. Il sistema deve supportare la firma digitale dei PDF generati con un certificato qualificato (eIDAS)? In caso affermativo, il CSI Piemonte fornisce un servizio centralizzato di firma, oppure il team deve integrare una soluzione dedicata?  | attualmente l'applicativo non prevede la generazione di pdf, è una funzionalità che prevedete di inserire nel TO-BE? |
